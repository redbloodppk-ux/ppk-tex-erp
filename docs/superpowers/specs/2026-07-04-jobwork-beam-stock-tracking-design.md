# Jobwork Beam Stock Tracking — Design Spec

## Background

Jobwork (as used in this mill) means: an external party supplies a ready-made
warp beam, the mill mounts it on its **own loom** and weaves it, then
delivers finished fabric back to the party via a DC and invoices the
job-work/weaving charge. This is the opposite direction from "outsource"
(where the mill sends its own beam OUT to a weaver who weaves off-site).

Beam-wise jobwork intake is already entered today on **Job Work → Warp beam
given** (`/app/jobwork`, `WarpBeamTab`), one row per physical beam in
`jobwork_warp_beam`. But those rows currently have no stock tracking at all:
`pavu_id`/`pavu_ids` are always null, and the beam number is stuffed into a
free-text `notes` field. These beams never appear in Loom View, can't be
mounted through the normal flow, and have no stock/status lifecycle.

Migration 230 previously added `pavu.production_mode = 'jobwork'` and
`pavu.jobwork_ledger_id`, plus a Pavu Master "Jobwork" tab with a manual add
form — but these were built on the wrong (outbound) assumption, mirroring
"outsource." They are being **repurposed**, not replaced, to represent the
correct inbound direction.

## Goal

When a jobwork beam-wise entry is saved on `/app/jobwork`, it should also
create a real `pavu` stock row tagged `production_mode = 'jobwork'`, so it
flows through the same stock → mount-on-loom → produce → finish lifecycle
that in-house beams already use — reusing Loom View, `AssignModal`,
`pavu_assign`, and the Beam Stock Report as-is.

## Schema change — migration 231

`pavu.sizing_job_id` is currently `NOT NULL REFERENCES sizing_job(id)`.
Jobwork-received beams have no sizing job (the mill didn't size them), so
this column must become nullable:

```sql
ALTER TABLE public.pavu ALTER COLUMN sizing_job_id DROP NOT NULL;
```

`pavu.jobwork_ledger_id` (added in 230) is reused as-is: it will now store
the `jobwork_party.ledger_id` of the party who **supplied** the beam (not who
it was sent to). Comment on the column will be updated to reflect the
corrected meaning. No other schema change needed — `beam_no`, `ends`,
`meters`, `production_mode`, `status` all already exist on `pavu` and cover
this case.

## `jobwork/page.tsx` — `WarpBeamTab.add()`, kind === 'jobwork' branch

For each beam row entered (`beamNo`, `ends`, `metres`):

1. Resolve `jobwork_party.ledger_id` for the selected party (same lookup
   pattern already used in the `kind === 'outsource'` branch).
2. Insert a `pavu` row: `sizing_job_id: null`, `production_mode: 'jobwork'`,
   `jobwork_ledger_id: <resolved ledger id>`, `beam_no`, `ends`, `meters`,
   `status: 'in_stock'`. `pavu_code` is auto-generated by the existing
   trigger, as always.
3. Insert the `jobwork_warp_beam` row as today, but set `pavu_id` to the
   newly created pavu row's id (single beam per row, so `pavu_ids` jsonb
   array stays unused/null — mirrors the existing single-beam pattern).

Existing restock/split/delete handlers in `jobwork/page.tsx` already contain
pavu-revert logic (flip `pavu.status` back to `in_stock`, then delete the
`jobwork_warp_beam` row). These need a quick check once `pavu_id` is
populated for jobwork rows, to confirm they behave correctly (revert status,
not hard-delete the pavu row, matching how outsource already handles this) —
no behavior change expected, just verification.

## Loom View (`pavu/assign/page.tsx`)

- Broaden the in-stock query from `.eq('production_mode', 'in_house')` to
  `.in('production_mode', ['in_house', 'jobwork'])`.
- Stock picker: jobwork-sourced beams get a small tag showing the supplying
  party's name (resolved via `jobwork_ledger_id` → `ledger` → party name,
  same join style already used elsewhere in this file).
- Mounted loom card: shows a "Jobwork — [Party]" tag when the mounted pavu's
  `production_mode = 'jobwork'`, analogous to how outsource-mode beams are
  (or would be) tagged.
- Remove the "Jobwork beams out with parties" read-only section added in
  commit `ffaa788` (wrong direction, now superseded by real stock
  integration).

## Pavu Master — Jobwork tab consolidation

The Jobwork tab's manual "add to jobwork party" cascade form (built last
session in `pavu-list-editor.tsx`) is removed. Beam-wise entry on
`/app/jobwork` becomes the single entry point for jobwork stock, consistent
with how it already works today for entering these beams at all. The tab
itself stays as a **view-only** list of jobwork-sourced pavu stock, now
reflecting real status (`in_stock` / `on_loom` / `finished`) instead of the
static read-only table.

`jobwork-beams-table.tsx` is updated to show real stock status per beam
(reusing `pavu.status`) instead of just a static list — replacing its
current "display-only, no stock" framing.

## Out of scope

- No changes to the outsource flow — it already works correctly for its
  (actually outbound) direction and is untouched by this repurposing.
- No changes to DC/invoice generation for jobwork parties — this spec is
  stock-tracking only, from intake to finished fabric; DC/billing already
  exists and is unaffected.
